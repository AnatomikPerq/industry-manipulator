# Анализатор EPLAN-схем на ошибки

Пайплайн: PDF -> базовые скрипты-парсеры -> папки с данными -> LLM-агенты -> JSON со списком ошибок.

## Запуск

```bash
pip install -r requirements.txt

python main.py --extract-only    # только парсеры PDF (LLM не нужны) - проверить извлечение
python main.py                   # весь пайплайн -> output/merged_report.json
python main.py --skip-extract    # только агенты, на уже извлечённых данных в data/
```

Из бэкенда (позже - с сайта):

```python
from main import run_pipeline
merged = run_pipeline()                                   # тип берётся из пометки в имени файла
merged = run_pipeline(doc_types={"файл.pdf": "scheme"})   # тип пришёл из формы загрузки
```

## Как это работает

1. Пользователь кладёт PDF в `data/base_files/`. **Имя файла должно начинаться с пометки типа:**
   - `(scheme)...pdf`  - принципиальная схема EPLAN
   - `(netlist)...pdf` - таблица соединений (нетлист)

   Файл без пометки или с неизвестной пометкой не обрабатывается: он попадает в
   `manifest.json` в `skipped_files` с причиной. Так гарантируется, что каждый базовый
   скрипт получает только документы своего типа. (Русские пометки `(схема)`, `(нетлист)`,
   `(таблица)` тоже понимаются - см. `TYPE_ALIASES` в `ingest.py`.)

2. `ingest.py` по пометке выбирает базовый скрипт из `data/base_analysis_scripts/`
   и складывает извлечённые данные в `data/<имя документа>/`:

   | тип | скрипты | что на выходе |
   |---|---|---|
   | `scheme` | `schematic_diagram_to_data.py`, `schematic_connectivity.py` | `raw/classified/graph/netlist/issues_candidates.json`, `nets.json`, `terminals.json` |
   | `netlist` | `netlist_to_json.py` | `connections.json` |

   На один тип документа может работать несколько скриптов - каждый достаёт свой срез.

   **Профили оформления** (`profiles.py`): форматные правила (regex'ы) вынесены по одному
   набору на шаблон проектного бюро, документ авто-детектится при загрузке:
   `A` Regul R500/KKS/EPLAN, `B` ОВЕН МВ210/МУ210, `C` Delta DVP, `D` ОВЕН 110.
   Без этого схема "чужого" бюро распознавалась почти пусто. Выбранный профиль пишется
   в `raw.json` и в статистику манифеста. Новое бюро = ещё один Profile, без правки пайплайна.

3. `data/manifest.json` - оглавление для агента: какой документ какого типа, где его
   данные, какие файлы получились и сколько в них чего.

4. **Стадия правил**: детерминированные чекеры, по одному на тип документа - без нейросети,
   за миллисекунды. Находки оформлены в том же формате `schema.py`, что и у агентов.

   | тип | чекер | что находит |
   |---|---|---|
   | `netlist` | `netlist_rules.py` | дубли физических адресов, тег в поле проводника, лишние суффиксы в KKS, незаполненные точки подключения |
   | `scheme` | `schematic_rules.py` | ссылка на несуществующий лист, межлистовая связь без ответной части |

   Правил для схемы намеренно мало: в обоих чекерах есть большой комментарий со списком
   проверок, которые **не** вошли, и с замерами, почему они дают ложные срабатывания
   (напр. "оборванная цепь" на реальных файлах оказывалась замкнутыми прямоугольниками
   и символами - 108 ложных находок на двух документах). Ложная находка дороже
   пропущенной: инженер идёт проверять её по чертежу и теряет доверие к отчёту.

5. Два LLM-агента (`oi_agent.py`, Open Interpreter) анализируют папку `data/` - каждый
   независимо, пишет и исполняет код, а не тащит файлы в контекст. Агентам оставлено то,
   что правилами не берётся: сверка таблицы подключений со схемой и формулировки. Свои
   вспомогательные скрипты они пишут в `data/your_helping_scripts_and_files/`.

6. Модель-сшиватель (`merge_reports.py`) объединяет два отчёта агентов, убирает дубли и
   вырезает ошибки из `known_errors.json` (заведомо не баги). Затем к результату
   **детерминированно** (без LLM) добавляются находки чекера - они ground truth и не
   должны потеряться на слиянии -> `output/merged_report.json`.

Каждый JSON-ответ модели валидируется по `schema.py`; при невалидном ответе модель
просят переисправить (`validation.py`, `agent.max_json_repair_attempts` в `config.yaml`).

## Папка data/ - песочница агента

```
data/
  manifest.json                   # оглавление (агент читает первым)
  base_files/                     # исходные PDF пользователя
  base_analysis_scripts/          # базовые скрипты-парсеры
  your_helping_scripts_and_files/ # рабочая папка агента
  <имя документа>/                # извлечённые данные по каждому документу
```

Агент видит всю `data/`: он может посмотреть, как парсер получил данные, и при
необходимости сам обратиться к исходному PDF.

## Что агент выдаёт на выходе

`output/merged_report.json`: `{"errors": [...], "summary": "..."}`. Структура одной находки
описана в `schema.py` (там же - два примера, которые уходят прямо в промпт модели).

Находки бывают ДВУХ РОДОВ, и обе лежат в одном списке `errors[]`:

- **`scope: "cross_document"`** - несостыковка МЕЖДУ документами (таблица подключений против
  схемы). В `refs[]` **ровно два** ref'а: первый `doc_type: "netlist"`, второй `doc_type: "scheme"`.
  Они и рисуют в таблице пользователя левую группу колонок ("Таблица подключений") и правую
  ("Монтажная документация"). Виды (`kind`): `MISMATCH`, `MISSING`, `REVIEW`.
- **`scope: "single_document"`** - ошибка ВНУТРИ одного документа. Один ref (или два ref'а на
  один и тот же документ, если это дубль). Правая группа колонок пустая. Виды (`kind`):
  `DUPLICATE`, `BROKEN_LINK`, `FORMAT`, `INCOMPLETE`.

Ref - это одно место в документе в доменных полях: `document`, `doc_type`, `source_file`,
`sheet`, `row`, `cabinet`, `terminal_block`, `pin`, `terminal_type`, `marking`, `kks`,
`conductor`, `found` ("что найдено на схеме"). Чего в документе нет - `null`.

Сверх ref'ов у находки есть: `kind`, `scope`, `severity` (critical/high/medium/low/info),
`type` (короткий подтип для группировки), `finding` ("что найдено" - факт по данным),
`action` ("что исправить или уточнить" - действие для инженера) и `evidence` (цитата из данных).

Колонку "Фрагмент монтажной схемы" из HTML-примера пока не делаем. Когда понадобится -
добавим в ref координаты (`sheet` + bbox), и сайт будет вырезать фрагмент из PDF сам,
отдельным детерминированным шагом, без ЛЛМ.

## Настройка

`config.yaml`: адреса/модели LLM-серверов (`llm_servers`), пути (`paths`), число попыток
авторемонта JSON (`agent`), `extraction.reuse_existing` - не перепарсивать PDF, если папка
с данными уже есть.
