# Анализатор проектной документации на ошибки

Пайплайн: PDF/XLSX -> базовые скрипты-парсеры -> папки с данными -> детерминированные
чекеры + LLM-агенты -> JSON со списком ошибок.

## Запуск

```bash
pip install -r requirements.txt

python main.py --extract-only    # только парсеры (LLM не нужны) - проверить извлечение
python main.py --rules-only      # только детерминированные чекеры + сверка связок, без LLM
python main.py                   # весь пайплайн -> output/merged_report.json
python main.py --skip-extract    # только агенты, на уже извлечённых данных в data/
```

Из бэкенда (с сайта):

```python
from main import run_pipeline
merged = run_pipeline()                                   # тип берётся из имени файла
merged = run_pipeline(doc_types={"файл.pdf": "scheme"})   # тип пришёл из формы загрузки
merged = run_pipeline(skip_agents=True)                   # режим "без ИИ": только скрипты
```

## СВЯЗКА - главное понятие

**Связка** - комплект документов ОДНОГО шкафа. Полная связка - это три взгляда на одно
и то же железо, которые обязаны друг другу не противоречить:

| марка | тип | что это |
|---|---|---|
| `Э3` | `scheme` | принципиальная схема - как всё соединено электрически |
| `СБ` | `assembly` | сборочный чертёж - как изделия размещены в шкафу физически |
| `СО` | `spec` | спецификация (xlsx) - что заказано, каким артикулом, в каком количестве |

Самые дорогие ошибки живут не внутри документа, а МЕЖДУ ними: изделие нарисовано в
шкафу, но не заказано; в спецификации один артикул, на чертеже другой; документ
скопирован из другого проекта и в штампе осталось чужое обозначение. Однодокументный
чекер такого не видит по построению - для этого есть стадия связок (`bundle_rules.py`).

Ключ сверки элемента между тремя документами - **позиционное обозначение**
(`designator`): `1QF1`, `DO1`, `G1`. Сверка идёт ТОЛЬКО внутри одной связки: разные
связки - разные шкафы.

Как определяется связка (по убыванию приоритета, см. `bundles.py`):

1. явно, аргументом `bundle_overrides={"файл.pdf": "связка 1"}`;
2. по подпапке в `base_files/`: всё в `base_files/связка 1/` - одна связка. Это
   единственный способ разобрать несколько проектов за один прогон, и это уже не
   догадка, а явное решение пользователя;
3. иначе **все документы прогона - ОДНА связка**: за один прогон грузят комплект
   одного шкафа.

**Разбиение по имени файла пробовалось и провалилось.** Ключ брался общим префиксом
до марки вида, и на следующем же наборе файлов комплект одного шкафа разъехался на
две «связки»:

```
026_812_19_..._АТХ2_015_СО_29_04_26_—_копия.xlsx  -> ключ "026_812_19_..._АТХ2_015"
СБ_ИК.3912-АТХ2.015_01.06.2026.pdf                -> ключ ""   (марка стоит ПЕРЕДИ!)
```

Причина фундаментальная: имя файла — не данные, а привычка бюро. Марка вида стоит
то в конце, то в начале; обозначение пишут то через точки, то через подчёркивания;
встречается «— копия». После такой ошибки сверка **молча не выполняется** — самый
дорогой сорт отказа. Не возвращайте эту эвристику.

### Связка в полном проекте

При загрузке альбома целиком (`full_project.py`) **связка = ШКАФ**, а не весь прогон:
в альбоме их до пятнадцати, и обозначения в них пересекаются (`1QF1` есть и в ШУПЧ1,
и в ЩС1, но это разные аппараты). Части раскладываются в `base_files/<шкаф>/`, то
есть тем же механизмом «подпапка = связка», что и выше — править `bundles.py` не
пришлось.

Спецификация и кабельный журнал идут на весь объект и шкафа в наименовании не
называют — они уходят в связку `общие документы`. Спецификация оттуда **одалживается**
каждой связке-шкафу (`_lend_project_wide_docs` в `main.py`) с пометкой `project_wide`,
которая выключает два направления сверки, заведомо ложных на общей спецификации:
`rule_spec_element_not_on_assembly` (в ней закономерно лежит оборудование чужих
шкафов) и `rule_designation_mismatch` (у неё обозначение альбома, а не шкафа).
По той же причине `spec_rules` пропускает `rule_duplicate_code`: один артикул в
нескольких строках здесь норма — замер дал 112 ложных находок на 837 строк.

## Как это работает

−1. **Нарезка альбомов** (`full_project.py`, до извлечения). Бюро отдаёт не комплект
   файлов на шкаф, а альбом целиком: 184–309 листов, внутри которых схемы десятка
   разных шкафов, общие виды, спецификация, кабельный журнал и планы расположения.
   Альбом опознаётся по числу листов (`is_full_project`, порог 80: самый большой
   одиночный документ корпуса - спецификация на 64, самый маленький альбом - 184),
   **переезжает** в `data/full_projects/` и режется на документы, которые ложатся в
   `base_files/<шкаф>/` с пометкой типа в имени (`(scheme)ЩС2. Схема ...pdf`). Дальше
   всё идёт обычным путём: «пометка типа» и «подпапка = связка» уже существуют.

   **Граница документа - графа «наименование» основной надписи** (ГОСТ 21.101). Часть =
   серия подряд идущих листов с одним наименованием; лист без графы (форма 4,
   продолжение) наследует наименование предыдущего. Два других сигнала **измерены и
   отвергнуты**: *номер листа* (на одном альбоме ведущая цифра `4.1…4.70` и есть номер
   документа, на другом нумерация сквозная и `.N` - подлист) и *форма штампа* (по ГОСТ
   верно, но одно бюро ставит полную форму на КАЖДОМ листе - 48 «документов», половина
   по одной странице). Оба проверяют привычку бюро, а не структуру альбома.

   Наименование **объекта** отделяется от наименования **документа** тем, что
   повторяется на >50% заполненных листов: в штампе они стоят в одной колонке друг под
   другом, и по координате их не разделить. Обозначения шкафов приводятся к кириллице
   (`_unify_layout`): бюро пишет однолинейную схему как `ЩC1` с ЛАТИНСКОЙ `C`, а
   принципиальную того же шкафа - как `ЩС1`, и без приведения это две разные связки,
   которые молча не сверяются. Планы расположения, молниезащита, установочные чертежи
   и ПЗ **сознательно не анализируются** (`TITLE_SKIP`) - комплектацию шкафа они не
   описывают.

1. Пользователь кладёт файлы в `data/base_files/` (можно плоско, можно по подпапкам-связкам).
   Тип документа определяется по имени файла:
   - **марка вида по ГОСТ 21.101** - `Э3` -> scheme, `СБ` -> assembly, `СО` -> spec.
     Файлы бюро уже названы так (`026.809.01.01-ИПК ША1 Э3_10.04.26.pdf`), поэтому
     руками ничего дописывать не нужно;
   - либо **явная пометка в начале имени**: `(scheme)...pdf`, `(netlist)...pdf`
     (русские `(схема)`, `(нетлист)`, `(спецификация)` тоже понимаются - см.
     `TYPE_ALIASES` в `ingest.py`).

   Файл, тип которого определить не удалось, не обрабатывается: он попадает в
   `manifest.json` в `skipped_files` с причиной. Так гарантируется, что каждый базовый
   скрипт получает только документы своего типа. Расширение тоже проверяется
   (`TYPE_SUFFIXES`): схему, чертёж и нетлист ждут в `.pdf`, спецификацию - в `.xlsx`
   либо в `.pdf` (в составе альбома книги Excel к ней не прилагается).

   **Альбом целиком** (`full_project.py`) разбирается ДО этого шага - см. стадию −1 ниже.

2. `ingest.py` по типу выбирает базовый скрипт из `data/base_analysis_scripts/`
   и складывает извлечённые данные в `data/<имя документа>/`:

   | тип | скрипты | что на выходе |
   |---|---|---|
   | `scheme` | `schematic_diagram_to_data.py`, `schematic_connectivity.py` | `raw/classified/graph/netlist/issues_candidates.json`, `nets.json`, `terminals.json` |
   | `netlist` | `netlist_to_json.py` | `connections.json` |
   | `assembly` | `assembly_drawing_to_data.py` | `assembly.json`, `assembly_raw.json` |
   | `spec` (.xlsx) | `specification_to_json.py` | `specification.json` |
   | `spec` (.pdf) | `specification_pdf_to_json.py` | `specification.json` |

   На один тип документа может работать несколько скриптов - каждый достаёт свой срез.
   Для спецификации скрипт выбирается **по расширению** (`scripts_by_suffix`): книга
   Excel и лист альбома - один и тот же документ по ГОСТ 21.110, но читаются
   принципиально разным кодом, и запускать оба на одном файле бессмысленно.

   **Сборочный чертёж** (`assembly_drawing_to_data.py`) НЕ читает векторную геометрию
   вообще: на одном листе её до 440 тысяч примитивов (контуры шкафа, перфорация коробов),
   и для поиска ошибок документации она бесполезна. Ценность чертежа - в подписях:
   у каждого изделия рядом стоит позиционное обозначение и артикул. Оказалось, что PDF
   сам группирует подпись в один текстовый блок (`'DO1'`(кегль 6.48) + `'DVP16SN11T'`(4.53)),
   и обозначение внутри блока набрано крупнее артикула - это и есть правило разбора.
   Такая пара помечается `pair_source="block"` (надёжно); если подпись разбита на разные
   блоки, артикул ищется ближайшим - `pair_source="nearest"` (ненадёжно, может быть чужой).
   **Сверка артикулов делается только по парам из блока** - см. чекер связки.

   **Спецификация** (`specification_to_json.py`) - колонки ищутся по тексту заголовков,
   а не по буквам (у ША1 их 16, у ШУ-ТМ - 9). Главное - развернуть колонку «Позиция» в
   отдельные обозначения: `1KL1...1KL3 2KL1...2KL5` -> 8 штук, `1KL1 ... 50KL1` -> 50
   (диапазон по ВЕДУЩЕМУ номеру). Разворот сделан сравнением двух концов диапазона по
   числовым полям, а не regex'ом "префикс+число" - только так одно правило покрывает обе формы.

   **Спецификация в PDF** (`specification_pdf_to_json.py`) отдаёт тот же
   `specification.json`, а разбор строки (раскрытие «Позиции», отсев разделов)
   **импортирует** из xlsx-парсера, чтобы две копии не разъехались. Колонки берутся из
   **линовки** таблицы: разделители нарисованы короткими отрезками по границам ячеек
   (длинных вертикалей на листе нет ни одной), поэтому копится суммарная длина
   вертикальных отрезков по каждому X. Границы по заголовкам **пробовались и не
   годятся**: колонки резко разной ширины, середина между «Позицией» и «Наименованием»
   проходит посреди наименования, и его перенос на вторую строку становился ложным
   обозначением (`4шт`, `248 Шайба 6.65Г.016 ГОСТ 6402-70`) - мусором в главном ключе
   сверки. Строки якорятся на колонке «Количество»: просвет по вертикали (шаг 23–30 px
   против высоты строки 14) не отличает соседнюю строку от переноса внутри ячейки.

   **Контуры букв выбрасываются** (`drop_glyph_hairlines` в
   `schematic_diagram_to_data.py`). CAD-экспорт кладёт текст на лист не только шрифтом,
   но и кривыми - обводкой каждой буквы, которая приезжает сотнями микроскопических
   отрезков НУЛЕВОЙ толщины. На листе 10.2 ЩСКЗ таких 7254 из 7423 линий, а проводов
   169. Признак - именно нулевая толщина: по длине отделить нельзя. Фильтр
   самоотключается, если штрихованных линий на листе почти нет, - бюро, которое чертит
   провода волосяной линией, не должно молча остаться без проводов вовсе.

   **Профили оформления** (`profiles.py`): форматные правила (regex'ы) вынесены по одному
   набору на шаблон проектного бюро, документ авто-детектится при загрузке:
   `A` Regul R500/KKS/EPLAN, `B` ОВЕН МВ210/МУ210, `C` Delta DVP, `D` ОВЕН 110,
   `E` щиты с номерными рейками (24-051-АК, ЩСКЗ), `F` Schneider Electric
   (Acti9/Zelio/Altivar). Без этого схема "чужого" бюро распознавалась почти пусто.
   Выбранный профиль пишется в `raw.json` и в статистику манифеста. Новое бюро = ещё
   один Profile, без правки пайплайна.

3. `data/manifest.json` - оглавление для агента: какой документ какого типа, **в какой
   связке**, где его данные, какие файлы получились и сколько в них чего. Раздел
   `bundles` - состав каждой связки и чего в ней не хватает.

4. **Стадия правил**: детерминированные чекеры, по одному на тип документа - без нейросети,
   за миллисекунды. Находки оформлены в том же формате `schema.py`, что и у агентов.

   | тип | чекер | что находит |
   |---|---|---|
   | `netlist` | `netlist_rules.py` | дубли физических адресов, тег в поле проводника, лишние суффиксы в KKS, незаполненные точки подключения |
   | `scheme` | `schematic_rules.py` | ссылка на несуществующий лист, межлистовая связь без ответной части, дубль адреса клеммы на листе, две катушки реле с одним обозначением |
   | `spec` | `spec_rules.py` | `#REF!` в ячейке, количество меньше числа позиций в строке, один артикул в нескольких строках |
   | `assembly` | `assembly_rules.py` | изделие подписано на одном листе чертежа и пропало с парного ему листа |

5. **Стадия связок** (`bundle_rules.py`) - сверка документов ОДНОГО шкафа между собой,
   тоже без нейросети. То, чего однодокументный чекер не видит по построению:

   | правило | что находит |
   |---|---|
   | `rule_designation_mismatch` | в штампах документов связки разные цифровые обозначения проекта (документ из чужого проекта) |
   | `rule_article_mismatch` | у одного обозначения в спецификации и на чертеже разные артикулы ("сверка характеристик") |
   | `rule_article_not_in_spec` | артикул подписан на чертеже, но такого изделия нет в спецификации - не заказано |
   | `rule_designator_not_in_spec` | обозначение есть И на чертеже, И на схеме, но строки в спецификации нет - не заказано |
   | `rule_spec_element_not_on_assembly` | изделие заказано, но его обозначения нет на чертеже (схема - третий голос) |

   Правил намеренно мало, и каждое опирается на однозначные строки (артикул, обозначение
   документа), а не на догадки о том, что и где должно быть нарисовано. В шапке каждого
   чекера - список проверок, которые **не** вошли, с замерами на реальных файлах:
   - "оборванная цепь" на схеме оказывалась замкнутыми прямоугольниками и символами -
     108 ложных находок на двух документах;
   - "обозначение есть на чертеже, но нет в спецификации" (по обозначениям) - на чертеже
     теми же буквами подписаны ВЫВОДЫ (`A1`/`A2` у катушки реле, `L1`, `COM`, `NO`), их
     сотни; поэтому проверка сделана по АРТИКУЛАМ, а не по обозначениям;
   - "количество меньше числа позиций" без оговорки `quantity > 1` - на ШУ-ТМ давало
     ровно одну находку, и та ложная (АВР-304: одно изделие на два выключателя `QF01, QF02`);
   - сверка артикулов по парам `pair_source="nearest"` - на ШУ-ТМ таких пар 3266 из 3277,
     артикул там сплошь притянут от соседа -> тысячи ложных "разные артикулы".

   Ложная находка дороже пропущенной: инженер идёт проверять её по чертежу и теряет
   доверие ко всему отчёту. Там, где догадка неизбежна (изделие не нарисовано - забыли
   или так и не рисуют?), находка оформляется как `REVIEW` - вопрос инженеру, а не
   утверждение об ошибке.

6. Два LLM-агента (`oi_agent.py`, Open Interpreter) анализируют папку `data/` - каждый
   независимо, пишет и исполняет код, а не тащит файлы в контекст. Агентам оставлено то,
   что правилами не берётся В ПРИНЦИПЕ: чекер сравнивает СТРОКИ, а агент понимает, ЧТО
   ЭТО ЗА ИЗДЕЛИЕ - должен ли вентилятор быть на схеме, соответствует ли «Автоматический
   выключатель NXB-63 3P 40А» подписи `C40A` на схеме. Плюс сверка таблицы подключений
   со схемой и формулировки. Свои вспомогательные скрипты они пишут в
   `data/your_helping_scripts_and_files/`.

7. Модель-сшиватель (`merge_reports.py`) объединяет два отчёта агентов, убирает дубли и
   вырезает ошибки из `known_errors.json` (заведомо не баги). Затем к результату
   **детерминированно** (без LLM) добавляются находки чекеров и сверки связок - они
   ground truth и не должны потеряться на слиянии -> `output/merged_report.json`.

Каждый JSON-ответ модели валидируется по `schema.py`; при невалидном ответе модель
просят переисправить (`validation.py`, `agent.max_json_repair_attempts` в `config.yaml`).

### Очередь к серверу ИИ

`run_pipeline` принимает `llm_gate` - функцию, которую зовёт ОДИН раз прямо перед
стадией агентов и которая может блокировать сколько угодно. Через неё веб-интерфейс
держит очередь: скрипты считаются сразу и параллельно с другими сессиями, а к
LM Studio пропускается строго одна. В CLI не используется - там прогон и так один.

### Сообщения о ходе разбора

`data/base_analysis_scripts/progress.py` печатает в stdout строки вида
`@@PROGRESS {"kind": "page", "page": 42, "total": 184}`. Их разбирает родительский
процесс (`web_app/queue_worker.py`) и показывает в интерфейсе, какой документ и какой
его лист читается прямо сейчас; в лог они не попадают. Канал выбран так, потому что
stdout между этими процессами уже есть и уже читается построчно, а парсеру не нужно
знать ни про сессии, ни про веб-интерфейс.

### Производительность

Скриптовая стадия целиком укладывается в секунды-минуты. Замеры (эта машина):

| что | время |
|---|---|
| комплект ЩСКЗ (схема 4 л. + чертёж 6 л. + спецификация) | **7 с** |
| альбом 24-051-АК: 309 листов -> 30 документов, 290 находок | **143 с** |

Главное, что стоит знать: `build_issue_candidates` в `schematic_diagram_to_data.py`
раньше занимал **98,6% всего времени разбора схемы** (34,7 с из 35,2 с на файле в
четыре листа; на альбоме - часы). Причина не в «медленном Python», а в том, что поиск
пересечений проводов был честным перебором ВСЕХ пар отрезков - 27 млн вызовов на
лист, - причём перебирались в основном **контуры букв**, пересекавшиеся сами с собой.
Отсюда же брались 13901 «пересечение» на файле, где проводов несколько сотен.

Исправлено двумя вещами, первая важнее: контуры букв выбрасываются
(`drop_glyph_hairlines`, n падает в ~44 раза), а пары перебираются по сетке bbox'ов.
Геометрия не тронута - решает тот же `_seg_intersect`. Итог: **34,7 с -> 0,02 с**,
пересечений 13901 -> 241, базовая линия ЩСКЗ по-прежнему ровно 6 находок.

Если снова покажется, что «скрипт висит, а процессор простаивает», - ищите такой же
квадратичный перебор, а не переписывайте на C.

## Папка data/ - песочница агента

```
data/
  manifest.json                   # оглавление + состав связок (агент читает первым)
  base_files/                     # исходные файлы пользователя (можно по подпапкам-связкам)
    связка 1/                     #   026.809.01.01-ИПК ША1 Э3/СБ/СО
    связка 2/                     #   026.808.01-ИПК ШУ-ТМ-14082-0002 Э3/СБ/СО
    ЩС1/ ЩС2/ общие документы/    #   у альбома - по подпапке на шкаф (нарезал сплиттер,
                                  #   метка .from_full_project)
  full_projects/                  # альбомы целиком; лежит РЯДОМ с base_files, а не внутри:
                                  #   base_files сканируется рекурсивно, и альбом внутри
                                  #   стал бы «связкой» из одного нечитаемого файла
  base_analysis_scripts/          # базовые скрипты-парсеры, чекеры, progress.py
  your_helping_scripts_and_files/ # рабочая папка агента
  <имя документа>/                # извлечённые данные по каждому документу
```

Агент видит всю `data/`: он может посмотреть, как парсер получил данные, и при
необходимости сам обратиться к исходному PDF.

## Что агент выдаёт на выходе

`output/merged_report.json`: `{"errors": [...], "summary": "..."}`. Структура одной находки
описана в `schema.py` (там же - два примера, которые уходят прямо в промпт модели).

Находки бывают ДВУХ РОДОВ, и обе лежат в одном списке `errors[]`:

- **`scope: "cross_document"`** - несостыковка МЕЖДУ документами. Два случая:
  - таблица подключений против схемы: в `refs[]` **ровно два** ref'а, первый
    `doc_type: "netlist"`, второй `doc_type: "scheme"`;
  - **связка** (спецификация / сборочный чертёж / схема): в `refs[]` **до трёх** ref'ов,
    по одному на документ, в порядке `spec` -> `assembly` -> `scheme`; документ, в котором
    элемента НЕТ, тоже получает ref (с `found: "не найден"`).

  Виды (`kind`): `MISMATCH`, `MISSING`, `REVIEW`.
- **`scope: "single_document"`** - ошибка ВНУТРИ одного документа. Один ref (или два ref'а на
  один и тот же документ, если это дубль). Виды (`kind`): `DUPLICATE`, `BROKEN_LINK`,
  `FORMAT`, `INCOMPLETE`.

Ref - это одно место в документе в доменных полях. Набор полей одинаков для всех типов
документов; поля делятся на две группы:

- «проводные» - `sheet`, `row`, `cabinet`, `terminal_block`, `pin`, `terminal_type`,
  `marking`, `kks`, `conductor`: ими описываются находки netlist <-> scheme;
- «элементные» - `designator` (позиционное обозначение - ключ сверки связки), `article`
  (артикул), `name`, `quantity`: ими описываются находки по связке scheme/assembly/spec.

Плюс общие: `document`, `doc_type`, `source_file`, `found` ("что найдено в этом документе").
Чего в документе нет - `null`.

В интерфейсе этим двум родам соответствуют ДВЕ таблицы (колонки у них разные и одной
таблицей не совмещаются): «Связка: спецификация, сборочный чертёж, схема» (колонки
Позиция / Артикул / Кол-во по каждому из трёх документов) и «Таблица подключений и схема»
(колонки Клемма / Штифт / Маркировка / KKS).

Сверх ref'ов у находки есть: `kind`, `scope`, `severity` (critical/high/medium/low/info),
`type` (короткий подтип для группировки), `finding` ("что найдено" - факт по данным),
`action` ("что исправить или уточнить" - действие для инженера) и `evidence` (цитата из данных).

## Фрагмент чертежа у находки (`fragment.py`)

У каждой находки в интерфейсе есть кнопка «фрагмент»: она показывает это место на самом
листе - вырезанный кусок PDF с обведённым попаданием.

Место ищется **поиском текста по самому PDF**, а не по координатам из находки.
Координат в ref'е нет и заводить их нельзя: ref описывает место В ДОМЕННЫХ ПОЛЯХ, и это
правильно - находки приходят и от чекеров, и от нейросети, а у нейросети координат нет
в принципе. Зато ключ находки (обозначение, артикул, адрес клеммы) напечатан на листе.

Что важно знать, если будете это править:

- **Ключи пробуются от точного к общему**: артикул -> обозначение -> маркировка -> KKS ->
  клеммник. Клеммник берётся БЕЗ номера вывода: адрес `1XT5:3` напечатан разорванным на
  фрагменты (ровно поэтому `schematic_rules` сшивает их обратно) и целиком не ищется.
- **Лист из находки - лишь первый кандидат.** Если ключа там нет, показывается лист, где
  он есть, но интерфейс ОБЯЗАН сказать об этом (заголовки `X-Fragment-Page` /
  `X-Fragment-Fallback`). На находках «изделие пропало с парного листа» отсутствие и
  есть суть, и молчаливая подмена листа выглядела бы опровержением находки.
- **Поворот листа.** Схемы приходят страницами с `/Rotate 270`, и `search_for` отдаёт
  координаты в НЕПОВЁРНУТОМ пространстве. Кадр строится по `rotation_matrix`, а рамки
  рисуются неповёрнутыми координатами (`draw_rect` пишет в поток содержимого страницы).
  Без этого на одних листах рендер падал, а на других - молча показывал не то место.

## Отчёт в PDF (`report_pdf.py`)

`output/merged_report.json` читает программа, таблица в браузере живёт до закрытия
вкладки. Для пересылки и подшивки к проекту отчёт собирается в PDF, и **начинается он с
описания анализа**: какие документы смотрели, какие связки сверяли, чем проверяли и -
главное - чего этот анализ НЕ проверяет. Без этой рамки список замечаний читается как
«вот все ошибки проекта», что неправда: часть проверок сознательно не делается, а
находки `REVIEW` - вопрос инженеру, а не утверждение об ошибке.

Вёрстка руками (курсор сверху вниз + разрыв страницы), шрифт - системный TTF: Base-14
Helvetica кириллицу отдаёт ненадёжно. Шрифт подмножествуется, иначе трёхстраничный
отчёт весит мегабайт.

## Настройка

`config.yaml`: адреса/модели LLM-серверов (`llm_servers`), число агентов (`agents.count`),
пути (`paths`, в т.ч. `full_projects_dir`), лимиты агента и число попыток авторемонта
JSON (`agent`), `extraction.reuse_existing`.

> `extraction.reuse_existing` сейчас управляет только тем, стирается ли папка документа
> перед разбором; сам разбор выполняется в любом случае. Раньше это было незаметно,
> потому что разбор схемы занимал полминуты на файл; теперь он занимает полсекунды.
