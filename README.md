# ⚙️ ИНДУСТРИЯ МАНИПУЛЯТОР

### Анализатор проектной документации

**Автоматическая проверка комплекта документов на шкаф управления**

Извлечение данных из PDF/XLSX → детерминированные чекеры → сверка связки →
**визуальный анализ схем нейросетью** → ИИ-агенты → отчёт.
Полностью офлайн, локальные LLM (LM Studio).

`V2.0` · Python 3.10+ · без облака

</div>

---

## Что это

Инструмент находит ошибки в комплекте проектной документации АСУ ТП: принципиальных
схемах (Э3), сборочных чертежах (СБ), спецификациях оборудования (СО), функциональных
схемах автоматизации (ФСА) и табличных документах — таблицах подключений, кабельных
журналах, перечнях сигналов и параметров.

Самые дорогие ошибки лежат **между документами**: изделие нарисовано, но не заказано;
у одного и того же аппарата в спецификации и на чертеже разные артикулы; в штампе
чужое обозначение. Поэтому документы одного шкафа проверяются не только каждый сам
по себе, но и друг против друга.

На выходе — таблица замечаний с точным указанием, **где** (лист, строка, шкаф, клемма,
позиционное обозначение, артикул), **что найдено** и **что требуется исправить**.
Отчёт можно выгрузить одним PDF, а место каждой находки — посмотреть фрагментом
самого чертежа, не открывая исходный файл.

## Два способа загрузки

| | Что грузите | Что получается |
|---|---|---|
| **Комплект на шкаф** | Несколько файлов: Э3, СБ, СО, нетлист | Одна связка — все документы описывают один шкаф |
| **Полный проект** | Один PDF-альбом на 180–300 листов | Альбом режется на документы, **каждый шкаф — своя связка** (их бывает до пятнадцати) |

Состав комплекта **не фиксирован**. Отсутствие какого-либо вида документа ошибкой
не считается и в отчёт не попадает — что сдавать, решает проектировщик.

## Из чего состоит

```
industry-manipulator/
├── web_app/              ← локальный веб-интерфейс: сессии, очередь, отчёт
│   ├── server.py         · бэкенд на стандартной библиотеке Python (без зависимостей)
│   ├── sessions.py       · сессия = комплект документов + её прогон + её отчёт
│   ├── queue_worker.py   · скрипты считаются сразу, к серверу ИИ — по очереди
│   └── static/           · страница интерфейса (HTML/CSS/JS, без сборки)
│
└── analyzer_to_errors/   ← пайплайн анализа (ядро)
    ├── main.py           · оркестратор всех стадий
    ├── full_project.py   · нарезка альбома на документы по штампу
    ├── ingest.py         · PDF/XLSX → структурированные данные
    ├── bundles.py        · какие документы описывают один шкаф
    ├── oi_agent.py       · ИИ-агенты (Open Interpreter)
    ├── report_pdf.py     · выгрузка отчёта в PDF
    ├── fragment.py       · фрагмент чертежа у места находки
    └── data/             · песочница: исходники, скрипты-парсеры, извлечённые данные
```

> Подробное устройство пайплайна, формат находок и схема данных —
> в [analyzer_to_errors/README.md](analyzer_to_errors/README.md).

## Быстрый старт

```bash
# 1. Зависимости
pip install -r analyzer_to_errors/requirements.txt

# адрес сервера ИИ — свойство вашей машины, а не проекта
cp analyzer_to_errors/config.local.example.yaml analyzer_to_errors/config.local.yaml

# 2. Веб-интерфейс
python web_app/server.py            # → http://localhost:8000
python web_app/server.py --port 9000
```

Дальше — в браузере:

1. **Создайте сессию** — это комплект документов одного шкафа (или один альбом).
2. **Перетащите файлы**: PDF для схем и чертежей, `.xlsx` для спецификации.
3. **Укажите тип** каждого документа. Обычно он определяется сам по марке вида
   в имени файла (`Э3`, `СБ`, `СО`, `NL`), но выбор в списке всегда главнее.
4. При необходимости раскройте **«Модели нейросетей для этой сессии»** и выберите
   модели агентов и зрения из тех, что подняты на сервере, а также число текстовых
   агентов (1 или 2). Пустой выбор означает «как в общем `config.yaml`». Настройка
   действует только на эту сессию и записывается в её `config.yaml`, поэтому потом
   всегда видно, чем считался отчёт.
5. **Запустите анализ**:
   - **Полный** (скрипты + ИИ) — извлечение, чекеры и ИИ-агенты;
   - **Без ИИ** — только извлечение и детерминированные чекеры. Быстро, ИИ не нужен.
6. Смотрите **отчёт** таблицей: у каждой находки есть кнопка «фрагмент» — она
   показывает это место на самом чертеже. Кнопка **«Скачать PDF»** выгружает отчёт
   файлом, с описанием проведённого анализа в начале.

Вкладку можно закрыть в любой момент: состояние сессии, лог и отчёт живут на диске.
Имя любого загруженного файла — ссылка, документ открывается в соседней вкладке.

### Очередь: ждёт только стадия ИИ

Скриптовая часть (извлечение и детерминированные чекеры) грузит локальный процессор,
поэтому запускается **сразу** и считается параллельно с чужими сессиями. В очередь
сессия встаёт **только перед стадией нейросетей** — LM Studio на всех один.
Режим «без ИИ» не ждёт вообще никогда.

Пока идут скрипты, интерфейс показывает, какой документ и какой его лист читается
прямо сейчас. Для стадии ИИ такого показателя нет и быть не может: агент сам решает,
какой файл открыть и в каком порядке.

### Без веб-интерфейса (CLI)

```bash
cd analyzer_to_errors
python main.py --extract-only    # только парсеры (LLM не нужны)
python main.py --rules-only      # только чекеры и сверка связок, мгновенно
python main.py                   # весь пайплайн → output/merged_report.json
python main.py --check-llm       # проверить серверы и модели ИИ
```

CLI сессий не знает и работает по общему `analyzer_to_errors/data/`.

## Требования

- **Python 3.10+**
- Зависимости пайплайна — `analyzer_to_errors/requirements.txt`
  (`openai`, `pyyaml`, `jsonschema`, `open-interpreter`, `pdfplumber`, `PyMuPDF`, `openpyxl`).
- Для полного анализа — сервер **LM Studio** с загруженными моделями.
  **Адрес задаётся в `analyzer_to_errors/config.local.yaml`** — это свойство конкретной
  установки, а не проекта, поэтому файл в репозиторий не едет (образец рядом:
  `config.local.example.yaml`, по умолчанию — эта же машина). Модели, лимиты и всё
  остальное берутся из общего `config.yaml`. Доступность проверяется прямо в интерфейсе
  кнопкой «Проверить серверы и модели ИИ».

Веб-серверу отдельные зависимости не нужны — он на стандартной библиотеке (`http.server`).

## Тесты

```bash
pip install -r analyzer_to_errors/requirements-dev.txt
pytest            # из корня репозитория, ~9 секунд
```

Ни PDF, ни сервера ИИ не требуют. Главное, что ими закрыто, — **золотые базлайны**:
находки чекеров на реальном корпусе (6 замечаний, разобранных по чертежу вручную) и
нарезка **трёх** настоящих альбомов (9, 49 и 67 документов — три разных бюро и четыре
формы основной надписи). Исходных PDF в репозитории нет — фикстуры собраны из уже
извлечённых данных (`analyzer_to_errors/tests/build_fixtures.py`), наименования листов
пересобираются командой `python tests/record_album_titles.py`.

Если базлайн упал — это не повод обновить эталон: находок стало больше, значит правило
начало выдавать ложные; меньше — перестало видеть настоящую ошибку.

### Главное правило: пропуск дороже ложного срабатывания

**Пусть анализатор лучше выдаст вдвое больше ложных находок, чем не найдёт хотя бы одну
настоящую.** Пропущенная ошибка не видна никому и никогда; лишняя видна сразу и
закрывается инженером за минуту — тем более что к каждой находке приложен фрагмент
чертежа. Поэтому сомнительное не гасится, а выдаётся как `REVIEW` — вопрос, а не
утверждение; и отказ нейросети или сервера тоже доезжает до отчёта, а не превращается
в тихое «ошибок нет».

## Как работает анализ (коротко)

| Стадия | Что делает | Нужен ИИ |
|---|---|---|
| **−1. Нарезка альбома** | Один PDF на 300 листов → отдельные документы по шкафам | нет |
| **0. Извлечение** | PDF/XLSX → JSON с данными по каждому документу | нет |
| **1. Правила** | Чекер каждого документа по отдельности | нет |
| **2. Связка** | Сверка документов одного шкафа друг против друга | нет |
| **2.5. Зрение** | Нейросеть смотрит на растр схемы и проверяет привязку подписей к линиям | да |
| **3. Агенты** | LLM-агенты анализируют данные, пишут и исполняют код | да |
| **4. Сшивание** | Объединение отчётов, удаление дублей и известных ошибок | да |
| **5. Валидация** | Каждый JSON-ответ модели проверяется по схеме с авторемонтом | да |

Находки чекеров — *ground truth*: они добавляются в итог детерминированно, уже после
работы моделей, и не теряются на сшивании.

**Правил сознательно мало.** Каждый чекер несёт список проверок, которые были
опробованы и **отвергнуты** по замеренной доле ложных срабатываний на реальных
файлах: ложное замечание обходится дороже пропущенного — инженер сверяется с
чертежом, ничего не находит и перестаёт доверять отчёту целиком.

## Отмена анализа

Анализ запускается отдельным процессом, поэтому отмена **мгновенная**: кнопка
«Отменить» убивает этот процесс целиком вместе со всеми потомками — эквивалент
Ctrl+C во всех его окнах разом, только гарантированный (сетевой вызов к ИИ или
Open Interpreter кооперативную отмену вполне мог бы проигнорировать).
Частичные результаты незавершённого прогона (`output/` и рабочая папка агента)
подчищаются автоматически; файлы пользователя и уже извлечённые данные остаются,
и сессию можно запустить заново сразу, ничего не перезагружая.

## Настройка

Всё в `analyzer_to_errors/config.yaml`: адреса и модели LLM-серверов (`llm_servers`),
число агентов (`agents.count`), пути (`paths`), лимиты и авторемонт JSON (`agent`),
`extraction.reuse_existing`.

Число одновременно считающих скриптовых воркеров — `SCRIPT_WORKERS` в
`web_app/queue_worker.py`. Слот к серверу ИИ там же и он **один**; поднимать его
можно только вместе с появлением второго сервера LM Studio.
